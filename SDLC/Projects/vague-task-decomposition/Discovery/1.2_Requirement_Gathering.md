# 1.2 Requirement Gathering: Vague Task Decomposition

## Functional Requirements
1. **Input**: Accept a single string representing a broad task.
2. **Decomposition**: The system must use an LLM to identify distinct, actionable steps.
3. **Approval**: The user must review and approve the list before any state changes occur.
4. **Execution**: Each approved subtask must be added to the `@default` list in Google Tasks.
5. **Feedback**: The system must provide a final summary of tasks created.

## Technical Constraints (May 2026 Standards)
1. **Model Specialization**:
   - Extraction: `Qwen 2.5 Coder 7B` (Maxwell 12GB).
   - Orchestration: `Qwen 3.6 35B` (Pascal 24GB).
2. **Quantization**: INT4 for weights, Q8_0 for KV Cache.
3. **Context**: Max 4k tokens for the extraction node.
4. **Tooling**: Must use the `gog` CLI. All commands must be absolute paths.
5. **Security**: `GOG_KEYRING_PASSWORD` must be injected from `Secrets/`.

## Jidoka Gates (Validation)
1. **Format Validation**: Ensure the LLM output is a parseable JSON array.
2. **CLI Validation**: Capture `gog` exit codes. Retry once if the error is a temporary network issue.
3. **State Verification**: Confirm the list of tasks is visible in the workspace (checked via `gog tasks find` or similar).

## Standardized Work Checkpoint
- [x] Are the hardware constraints identified? (Maxwell/Pascal)
- [x] Are the security requirements defined? (Secrets/)
- [x] Is the validation strategy clear? (JSON + CLI exit codes)
