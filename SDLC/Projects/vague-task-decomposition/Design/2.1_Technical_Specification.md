# 2.1 Technical Specification: Vague Task Decomposition

## Architectural Topology
- **Host**: Alienware (Windows/WSL2/Linux depending on environment, but using `ssh` protocols).
- **Inference**: 
  - **Supervisor**: Qwen 35B on Pascal (GPU 0).
  - **Worker (Extractor)**: Qwen 7B on Maxwell (GPU 1).
- **Quantization**: INT4 weights, Q8_0 KV Cache.
- **Context Limit**: 4,096 tokens (Worker).

## Node Specifications

### 1. `NODE_EXTRACT` (Skill: `llm-extract-action-items`)
- **System Prompt**: "You are an atomic task decomposition node. Input: raw text. Output: ONLY a JSON array of strings representing actionable subtasks. No conversational text."
- **Model**: `ollama/qwen2.5:7b` (pinned to Maxwell).

### 2. `NODE_TASK_ADD` (Skill: `google-tasks-create-task`)
- **Execution**: DIRECT execution of `gog` CLI.
- **Directive**: NO SHELL SCRIPT WRAPPERS. All commands must be called directly via the `exec` tool.
- **Hardening**: Must capture stderr. If `401 Unauthorized`, trigger re-auth protocol.

## Jidoka Gate Logic
- **Gate 1 (Regex/JSON)**: If output is not a valid JSON array, retry `NODE_EXTRACT` with error feedback.
- **Gate 2 (Empty Check)**: If array is empty, fail with "No actionable items found".
