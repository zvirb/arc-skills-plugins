# 6.3 KV Cache Management

## Objective
To optimize VRAM consumption and prevent "Cache Thrashing" in multi-agent orchestration on legacy Pascal/Maxwell hardware.

## Management Strategies

### 1. Model & Cache Quantization (Hardware Standard)
- **Model Standard**: All models MUST be deployed using **INT4 quantization** (GGUF/AWQ).
- **Cache Standard**: Use **Q8_0** KV cache quantization for the Orchestrator (Pascal) to stabilize long-context performance.
- **Benefit**: Mitigates PCIe 3.0 bottlenecking by reducing total VRAM throughput requirements.

### 2. Context Capping (GPU Role Pinning)
- **Orchestrator (Pascal - 24GB)**: Hard-limit context (`OLLAMA_NUM_CTX`) to **32,768 tokens**.
- **Specialist (Maxwell - 12GB)**: Hard-limit context to **4,096 tokens**.
- **Reason**: Prevents "Sudden Context Overflow" and ensures the Maxwell node remains available for high-frequency vision/micro-agent tasks.

### 3. Parallelism Constraints
- **Standard**: Set `OLLAMA_NUM_PARALLEL=1` and `OLLAMA_NUM_GPU=1` per instance.
- **Reason**: **CRITICAL:** Forbidding Tensor Parallelism is mandatory to prevent PCIe bus saturation on legacy dual-GPU nodes.

### 4. Cache-Aware Routing
- **Standard**: Map related tasks to the same agent session to leverage prefix caching.
- **Benefit**: Reduces "Prefill Latency" and avoids redundant context ingestion.

## Jidoka Rule: Automated Flush
If the KV cache fragmentation exceeds 20%, or if a `SessionWriteLockTimeoutError` is detected, the OpenClaw gateway should perform an automated **Cache Flush** and restart the model instance.
