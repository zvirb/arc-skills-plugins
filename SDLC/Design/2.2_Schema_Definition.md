# 2.2 Schema Definition

## Objective
To eliminate "vibe-based" engineering by enforcing strict, validated data contracts (The Contract Principle).

## Schema Standards
All I/O for OpenClaw nodes must be defined using **JSON Schema (Draft 7)** or **Pydantic**.

### 1. Input Schema (The Request)
- Must include all required parameters for tool execution.
- Must include optional parameters with default values.
- Must use `description` fields to guide the LLM on parameter semantics.

### 2. Output Schema (The Response)
- Must include a `status` field (e.g., `success`, `failure`, `retry`).
- Must include a `data` object for valid results.
- Must include an `error` object for failures, containing `code` and `message`.

## Schema-Guided Reasoning (SGR)
The agent must be designed to output its "thoughts" within a structured JSON wrapper.
```json
{
  "reasoning": "Determining if the PDF contains chemical diagrams.",
  "plan": ["Scan page 1", "Search for SMILES patterns"],
  "tool_call": { "name": "deepseek-ocr", "args": { "path": "/data/p1.png" } }
}
```

## Jidoka Validation
Nodes must programmatically validate their output against the schema.
- **Strict Mode**: If the output does not match the schema, the node must trigger an automatic retry loop (max 3) with the validation error fed back into the context.
