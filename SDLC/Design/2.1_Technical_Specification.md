# 2.1 Technical Specification

## Objective
To provide a rigorous, non-monolithic description of the agent system's architecture, following the "Spec First" protocol.

## Specification Components
Every OpenClaw technical spec must include:

### 1. Architectural Topology
- **Orchestrator Role**: Define the high-level logic (e.g., Qwen 3.6 35B on Pascal).
- **Specialist Roles**: Map atomic nodes to specific worker models (e.g., DeepSeek-OCR on Maxwell).

### 2. State Machine Logic
- Define how the system transitions from one state to another.
- Use Mermaid diagrams to visualize the "Standardized Work" flow.

### 3. Data Flow Persistence
- Specify where temporary state is stored (e.g., `Memory/`, `state/`).
- Define the lifecycle of `$step_id.json` artifacts.

## Design Constraints
- **Isolation**: No shared memory between non-parent/child nodes.
- **Statelessness**: Every atomic node must be capable of execution from a cold start using only its input schema and current filesystem state.

## Verification Checklist
- [ ] Is the topology optimized for the Maxwell/Pascal VRAM split?
- [ ] Are all state transitions deterministic?
- [ ] Is there a clear "Andon" loop for every external interaction?
