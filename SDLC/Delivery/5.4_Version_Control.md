# 5.4 Version Control & Rollback

## Objective
To ensure that every change to the agent's behavior is traceable and that the system can quickly return to a stable state (Jidoka).

## Versioning Rules

### 1. Behavior as Code
- Every change to a prompt, skill metadata, or plugin logic must be committed to Git.
- Use descriptive commit messages: `feat(skill): add retry logic to deepseek-ocr`.

### 2. The Configuration Freeze
- Before a major deployment, create a Git Tag (e.g., `v2026.05.05-stable`).
- Back up the current `openclaw.json` and `Secrets/` to the `Archive/` directory.

## Rollback Protocol
If the Jidoka validation fails post-deployment:
1.  **Revert Git**: `git revert <commit_hash>`.
2.  **Restore Config**: Restore the `openclaw.json` backup from `Archive/`.
3.  **Re-Install**: Run `openclaw skills install` for the previous version.
4.  **Verify**: Run the "Heartbeat" task to confirm stability.

## Jidoka Check
- [ ] Is the current production hash identical to the `main` branch?
- [ ] Is there a valid rollback artifact in `Archive/` for the current deployment?
