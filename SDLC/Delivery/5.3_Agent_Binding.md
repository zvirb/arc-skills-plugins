# 5.3 Agent Binding & Capability Injection

## Objective
To configure the agent's identity and capabilities by mapping slugs to roles in `openclaw.json` while maintaining deterministic tool discovery.

## The Binding Contract
An agent is defined by its `profile` and its `skills`. 

### 1. Capability Inheritance
- **Explicit Binding**: Manually add the skill slug (e.g., `google-tasks-manager`) to the `skills` array. 
- **CRITICAL**: Never rely on auto-registration ("Ghost Skills"). Every capability must be explicitly allowlisted to prevent context bloat and unauthorized tool execution.

### 2. Static vs. JIT Discovery
- **Deterministic (Lobster)**: For agents executing .lobster workflows, use **Static Binding**. Explicitly declare tools in `tools.alsoAllow` to bypass the probabilistic JIT `tool_search`.
- **Reactive (Chat)**: Use the `tool_search` mechanism for high-latency, broad-capability chat sessions. This reduces KV cache thrashing by loading tools only on-demand.

### 3. Identity & Role Pinning
- **Standard**: Pin Maxwell (12GB) endpoints for vision/workers and Pascal (24GB) for orchestrators.
- **Context Engines**: If using a custom skill management engine, set `kind: "context-engine"` in the plugin manifest.

## Zero Trust Configuration (CRITICAL)
- **Rule**: Agents are strictly FORBIDDEN from writing to `openclaw.json`. 
- **Reason**: Writing to the configuration file via an agent can sever GitOps symlinks and cause "Redaction Lockout," where real API keys are overwritten by `***` placeholders.

## Verification Gate
After binding:
1.  **Stop Gateway**: `systemctl stop openclaw-gateway`.
2.  **Verify Manifest**: Ensure `openclaw.json` matches the Git source of truth and contains no `***` redactions.
3.  **Start Gateway**: `systemctl start openclaw-gateway`.
4.  **Confirm Role**: Verify the agent identifies as the correct persona and can see its bound tools.
