# 4.1 Deterministic Evaluation

## Objective
To define the "Master Success Criteria" that eliminate ambiguity in testing (Standardized Work).

## Evaluation Rules

### 1. The Schema Match
Success is defined **only** by the presence of valid, schema-compliant data. Conversational confirmations ("I have done the task") are ignored.

### 2. Physical Traceability
Every success must leave a physical trace in the environment:
- A new file in `Memory/`.
- A new entry in `Logs/`.
- A changed state in an external API (verified via a **separate** tool).

### 3. Verification Protocol (CRITICAL)
- **Observer Only**: Verification tools (e.g., `list`, `get`) must ONLY observe state. They are strictly FORBIDDEN from modifying or completing tasks.
- **Independent Confirmation**: Always use a separate tool for verification (e.g., use `browser` to verify a `gog` calendar creation). 
- **No Self-Healing**: If a verification tool discovers an issue, it must report it as a failure. Do NOT use the verification tool to "fix" the state.

### 4. Bit-Perfect Output
For data transformation nodes, success is binary. Use checksums or exact-match JSON comparisons to verify that the transformation logic was followed perfectly without hallucination.

## Evaluator Node Pattern
For every tool-calling node, a corresponding Evaluator node must be implemented:
- **Node A (Actuator)**: `gog calendar create`.
- **Node B (Evaluator)**: `browser navigate calendar.google.com` (Independent physical confirmation).
