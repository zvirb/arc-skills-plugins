# 3.1 Atomic Node Construction

## Objective
To build single-responsibility, highly testable units of work (Standardized Work) within a deterministic execution graph.

## Implementation Rules

### 1. The Single File Rule
An atomic node should ideally be contained within a single file (e.g., `SKILL.md` or a specific TypeScript function). Complex logic must be moved to **Plugins** to leverage Node.js worker thread offloading.

### 2. Deterministic Orchestration (Lobster)
- **Standard**: Avoid probabilistic LLM tool loops for multi-step tasks. Use **.lobster** macros to define unyielding, typed pipelines.
- **Rule**: Every .lobster workflow must have a unique `id` and clearly defined `args` with schema validation.

### 3. The Payload Handoff Protocol
- **Standard**: Do NOT pass large strings (>10KB) via JSON-RPC parameters.
- **Protocol**: Use the **PluginArtifact** schema for inter-step communication.
- **Heavy Payloads**: Use the **lobster://** protocol (Blob storage) for massive data (PDFs, multi-MB logs).

### 4. Session Isolation (Concurrency)
- **Standard**: For specialist agents in a pipeline, use **strictly distinct, isolated sessions**.
- **Reason**: Sharing a single session ID triggers `SessionWriteLockTimeoutError` and destroys determinism.

## Node Archetypes

| Archetype | Primary Function | Example Tool / Macro |
| :--- | :--- | :--- |
| **Fetcher** | Retrieves data from the environment. | `fs-read`, `lobster://` streaming. |
| **Transformer** | Maps data via PluginArtifact. | `pick`, `where`, `jq`. |
| **Actuator** | Executes an external action. | `gog tasks create`, `exec`. |
| **Evaluator** | Validates the success of an action. | `fs-exists`, `diff.last`. |

## Jidoka Validation
Every node must include an internal `try-catch` block that returns structured JSON. In .lobster, utilize **Approval Gates** (`approval: required`) for high-impact side effects.
